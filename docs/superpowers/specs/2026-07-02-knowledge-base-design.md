# Knowledge Base: MD Import, Ask (Full RAG), UI, Eval - Design Spec

**Date:** 2026-07-02
**Status:** Approved (brainstorm 2026-07-02)
**Type:** Feature spec (single implementation plan expected)
**Relates to:** `2026-06-17-rag-roadmap-and-real-project-pipeline-design.md` (roadmap). This work turns the sandbox into a usable knowledge base and lays the metadata foundation for roadmap step 2. Roadmap ladder (metadata filtering, query transform, Graph RAG) stays valid and unchanged.

---

## 1. Goal

Turn the springboot-rag sandbox into a project knowledge base:

- Import markdown files (upload), chunk them structure-aware, embed and store with metadata.
- Semantic search over them (existing backends, unchanged).
- Ask questions and get answers generated by a local LLM from retrieved chunks (full RAG loop).
- Simple web UI for import / search / ask.
- Java-native evaluation harness measuring retrieval quality and answer faithfulness.

### Decisions locked during brainstorm

| Decision | Choice | Notes |
|----------|--------|-------|
| Scope | Full RAG (retrieve + generate answer) | Not retrieval-only |
| Corpus | Mixed md: project docs/specs, technical docs with code, incident/ops notes, meeting notes | Meeting notes rare |
| Chunking | Markdown-aware first; `Chunker` becomes interface | Semantic chunker = later comparable experiment, not now |
| Eval | Java-native (golden set + local LLM judge) | RAGAS parked for later (Python client calling Java API, when a strong judge model such as Azure OpenAI is available). Local small models are too noisy as judges. |
| Architecture | Extend sandbox in place (Approach A) | No new module, no real-project pipeline jump |
| UI | Single static page served by Spring Boot | Vanilla JS, no build toolchain |

## 2. Architecture

```
POST /documents (md file upload)
   +-> MarkdownChunker -> chunks with metadata {source_file, heading_path, position}
         +-> existing embed + store (pg chunks table + Qdrant payload gain metadata)

GET /ask?q=...
   +-> SearchService (hybrid + rerank, existing) -> top-k chunks
         +-> ChatProvider.answer(question, chunks) -> answer + cited chunk refs
```

Invariant rule from the roadmap holds: every new capability = new class behind an interface. Existing `SearchService`, storage, fusion, rerank code is not rewritten. `/compare` is untouched.

## 3. Components

### 3.1 Chunker interface + MarkdownChunker

- `Chunker` becomes an interface: `List<Chunk> chunk(String docId, String text)` where `Chunk` carries `text`, `headingPath`, `position`. Current implementation renamed `WordWindowChunker`, kept working (used as fallback and still available for comparison).
- `MarkdownChunker` (new, default for md ingest):
  - Parse with **commonmark-java** (`org.commonmark:commonmark`) AST walk - NOT regex. New dependency, approved 2026-07-02 (~200 KB, zero transitive deps).
  - Split on heading boundaries.
  - Fenced code blocks and tables are atomic - never split mid-block.
  - Each chunk text is prefixed with its heading breadcrumb, e.g. `# Doc Title > ## Section > ### Subsection`.
  - Size cap ~400 tokens (word-count proxy is fine); an oversized section falls back to the word-window strategy inside that section, breadcrumb preserved on every piece.

### 3.2 Document upload + management

- `POST /documents` - multipart upload of one `.md` file.
  - docId = sanitized filename; re-upload with same name replaces (existing delete-then-insert upsert in `IngestService`).
  - Constraints: `.md` only, max ~2 MB, UTF-8 only. Violations -> 400 via existing `GlobalExceptionHandler` pattern.
  - Response: docId + chunk count.
- `GET /documents` - list ingested docs (docId + chunk count); one query on existing chunk table.
- `DELETE /documents/{docId}` - reuses existing `IngestService.delete`.
- Existing JSON ingest endpoint stays unchanged.

### 3.3 Metadata

- New columns on the pg chunk table + Qdrant payload fields: `source_file`, `heading_path`.
- Stored on ingest, returned in search hits and `/ask` sources.
- Filtering on them is NOT in scope here - that is roadmap step 2 proper. This work only lays the storage foundation.

### 3.4 ChatProvider + OllamaChatProvider

- `ChatProvider` interface, same swap-to-Azure pattern as `EmbeddingProvider`.
- `OllamaChatProvider` calls the Ollama chat API (model configurable, e.g. `qwen2.5` / `llama3.1`).
- Config properties follow the existing `EmbeddingProperties` pattern.

### 3.5 AskService + GET /ask

- Pipeline: retrieve via best existing path (hybrid + rerank) -> top-k chunks -> prompt LLM -> `{answer, sources[]}`.
- Prompt shape: system prompt instructs "answer only from provided context, cite chunk numbers, say 'not found in knowledge base' if context lacks the answer" + numbered chunks + question.
- Answer text cites chunk indices; response maps them to source metadata (docId, headingPath) so the UI can render citations.
- Short-circuit: if retrieval returns nothing (empty knowledge base), return "knowledge base empty" without calling the LLM.

### 3.6 UI

Single static page in `src/main/resources/static/` - one HTML + one JS + one CSS file. Vanilla JS + fetch. No framework, no build step.

Three zones:

1. **Import** - drag/drop or file picker for `.md` -> `POST /documents`, shows chunk count. Document list with delete buttons (`GET /documents`, `DELETE /documents/{docId}`).
2. **Search** - query box + backend dropdown (fts / vector / qdrant / hybrid / rerank) -> existing `/search`, results show chunk text + heading path + score.
3. **Ask** - question box -> `GET /ask`, renders answer + collapsible source chunks with heading breadcrumbs.

### 3.7 Eval harness (Java-native)

- Golden set: `src/test/resources/eval/golden.yaml`, ~20-30 entries of `{question, expectedDocId, expectedHeadingPath (optional)}` hand-built from real project docs.
- Retrieval eval: JUnit test tagged `eval`, excluded from the normal build, run manually. For each backend, computes and prints a table of:
  - recall@5 (expected doc in top 5)
  - MRR (mean reciprocal rank of first correct hit)
  - hit-rate@1
- Faithfulness eval: separate tag `eval-judge` (slow, needs Ollama chat model up). For each golden question: run `/ask` pipeline, then one judge prompt to the local LLM: "is this answer fully grounded in these chunks - yes/partial/no". Report counts. Known limitation: local judge is noisy; treat as smoke signal, not ground truth. RAGAS with a strong judge replaces this later.

## 4. Error handling

| Case | Behavior |
|------|----------|
| Non-md, oversized, or non-UTF-8 upload | 400 with message (existing `GlobalExceptionHandler`) |
| Ollama chat model missing / down | 503 "chat model unavailable"; embedding/search paths unaffected |
| Empty knowledge base on `/ask` | "knowledge base empty" response, LLM not called |
| Qdrant/pg failures on ingest | existing behavior unchanged (IllegalStateException -> 5xx) |

---

## 5. Testing

- `MarkdownChunkerTest` - pure unit: heading split, code block atomicity, table atomicity, breadcrumb prefix, oversize fallback, empty/blank input.
- `AskService` unit test with fake `ChatProvider` - prompt assembly, citation mapping, empty-retrieval short-circuit.
- Upload/list/delete endpoint tests via Testcontainers (existing pattern).
- Eval harness as JUnit tags `eval` / `eval-judge`, excluded from normal build.
- UI: manual verification only - no JS test toolchain for 3 forms.

---

## 6. Out of scope (YAGNI, roadmap unchanged)

- Metadata **filtering** API/UI (roadmap step 2 proper) - only storage foundation here.
- Query transform (step 3), Graph RAG (step 4).
- Semantic chunking (later comparable experiment behind the `Chunker` interface).
- RAGAS eval (later, Python client + strong judge model).
- Auth, multi-tenancy, chat history, PDF parsing.
